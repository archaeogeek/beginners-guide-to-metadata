Metadata is usually described as data about data, but that doesn't really tell you much. Think of it as the difference between heating up some custard to go on your pudding, or some dog food.

But what is it really?

Additional information that sits alongside the dataset it relates to

It should contain a standard set of information, in a standard format, so it's both human and machine-readable

It can be stored in a separate file, in a catalogue, or embedded in the dataset itself. We're going to focus on using the GeoNetwork metadata catalogue because it allows us to do all of this, in an open source, interoperable way

Why is it important?

The direct benefit: It helps people find and acquire the right data for their task

Why is it important (part 2)?

Many indirect benefits: Improved data sharing increases efficiency and reduces cost

Key Terminology

XML

eXtensible Markup Language

The most common format in which metadata is provided

Both a language and a file-format

Takes the form of elements, surrounded by start and end tags that describe what the element is and what type of thing it is

XML continued

Rules on how and what should be in a metadata record are defined by schemas

A record can contain elements from different shared vocabularies, using XML namespaces

A valid record must be both "well-formed" (eg no errors in the xml structure) and "schema-valid"

Validation rules are often supplied as schematron files

We probably want to show an example of an xml record with various bits highlighted

Related terms

XSL(T): used for transforming XML documents into other XML, or even other formats such as HTML, text

XPath: a way of referring to elements in an XML document

XQuery: a SQL-like way of querying the elements in an XML document

These are used extensively within metadata catalogues such as GeoNetwork to control the display of records on the page, and to allow updates to records. We probably want to show an example of an xml record with various bits highlighted

Schemas, standards and profiles

Schema: specifies the structure of the metadata, the elements, the vocabularies

Standard: a schema that has been developed and maintained by a standards organisation

Profile: an extension to a standard, or an implementation of it in software, or both

You'll hear all of these used interchangeably when people talk about metadata. Generally speaking, a profile can only be more strict than the standard it extends. This might be by making an element mandatory when the core standard says it's optional, or by adding a controlled vocabulary. Elements from other standards can be added as long as they don't conflict.

ISOwtf and Gemini

Standards start as a set of rules defined "on paper", and the core standard for Geospatial metadata is ISO19115

ISO19139 is the implementation of that paper standard in XML

GEMINI is an extension to ISO19139 for UK geospatial metadata

Gemini adds in some UK-specific terms and code lists, reduces some choices (such as language), and adds some elements from other standards such as ISO19119 for services

But what about INSPIRE?

Do we still need it? Yes!

INSPIRE had a really good purpose- to make it easier to share datasets across geographic and political boundaries

Literally all you have to do is make metadata available for a few key datasets, in UK Gemini format, and make it, and the data, discoverable

The INSPIRE regulations were amended in 2018 so they could be retained in UK law after Brexit. The original reasoning for INSPIRE was interoperable cross-border environmental data- so you could stitch together (for example) flood model data over country and county boundaries and have it match up. What's not to like about that?

Other standards and acronyms

Dublin Core is a basic, domain-agnostic standard for describing any sort of resource

DCAT and DCAT-AP are profiles of Dublin Core, widely used in catalogues such as CKAN

Rather than XML, Dublin Core and DCAT use RDF (Resource Description Framework)

The Dublin in Dublin Core relates to Dublin, Ohio rather than Dublin, Ireland. RDF is a data model that comprises "triple statements", comprising a subject, a predicate or relationship, and an object, and it's widely used for linked data and semantic web implementations. A good metadata catalogue will allow you to publish and ingest metadata in both ISO19139 and DCAT formats.

How do I even start with all of this

Do as little as possible!

Use a standard template for your metadata and you can import it into a catalogue when you're ready

Automatically create metadata where possible

Use open standards for data and metadata to make sharing easy

Use an open source catalogue to get help, bug fixes, and new developments

So if I don't do spatial, can I just switch off?

No! Use it for non-spatial datasets, and workflows, and even GDPR reporting

The core ISO19139 standard allows the use of elements from many other standards. A good metadata catalogue will allow you to do a deep dive into these elements to focus on what's important
